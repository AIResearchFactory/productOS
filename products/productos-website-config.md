# productOS Website Configuration

## Purpose
This document defines the full website structure, page goals, content blocks, and design intent for productOS.

This version is intentionally broad. It presents productOS as a system for connecting AI tools, running research and analysis, creating recurring workflows, and generating operational outputs across many use cases, not just one narrow example.

---

# 1. Site Strategy

## Core positioning
productOS is an AI operating system for real work.

It helps people and teams:
- connect their preferred AI tools
- create structured projects
- run research and analysis
- automate recurring workflows
- generate outputs like docs, PRDs, and strategy artifacts
- switch across providers such as Ollama, Codex, Claude, and Gemini

## Key website goals
- explain what productOS is quickly
- make the product feel real and operational
- show breadth of use cases without feeling vague
- demonstrate flexibility across providers and workflows
- drive users toward install / try / demo / waitlist / GitHub

## Tone
- practical
- product-led
- not hypey
- clear and modern
- technical enough for serious users, but still understandable

## Design direction
- dark, structured, modern
- dense enough to feel powerful
- screenshots or UI states where possible
- strong information hierarchy
- modular sections with clear CTAs

---

# 2. Global Content Principles

## Every page should do one clear job
No page should try to explain the whole product at once.

## Show system behavior, not abstract promises
Prefer:
- create a project
- run research
- set a workflow
- generate a PRD

Over:
- unlock AI productivity
- transform your business

## Keep provider flexibility visible
Make it clear the system can work with:
- Ollama
- Codex
- Claude Code
- Gemini

## Emphasize operational outputs
Show real outputs like:
- research briefs
- competitor analysis
- daily reports
- workflows
- PRDs
- internal docs

# 8. Notes for Design / Dev

## Visual system
- dark modern interface
- real product screenshots preferred
- minimal decorative fluff
- strong card hierarchy
- visible project/workflow context

## Technical notes
- design homepage for easy screenshot updates
- structure pages modularly so screenshots and copy can evolve
- example project page should reuse same blocks as other pages where possible

---

# 9. Final Recommendation

Do not build the website around one narrow story.

Instead:
- explain the full product clearly
- show multiple workflow types
- include one concrete example project walkthrough
- keep screenshots real and operational

That gives you both breadth and credibility.
